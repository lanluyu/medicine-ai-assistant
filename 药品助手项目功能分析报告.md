## 一、项目概述

本项目是一套基于 **RAG（检索增强生成）** 技术的药品信息智能问答系统，以 10 万条药品说明书为知识库，通过大语言模型提供自然语言问答服务。

### 典型查询场景

| 场景 | 示例问题 |
|------|----------|
| 按药品名称查询 | "阿莫西林的用法用量是什么？" |
| 按症状推荐用药 | "感冒发烧头痛吃什么药？" |
| 复杂用药咨询 | "高血压患者感冒了可以吃哪些感冒药？" |
| 禁忌/副作用查询 | "哪些感冒药含有嗜睡成分？" |
| 特殊人群用药 | "儿童发烧推荐哪些退烧药？" |

### 核心设计原则

- **llama-index** 负责向量化与检索（离线预计算，在线毫秒级召回）
- **LangChain + LangGraph** 负责 Agent 推理与多轮对话管理
- **Qwen3.5-plus** 作为最终回答生成模型（阿里云 DashScope API）
- **FastAPI + SSE** 提供类 ChatGPT 的流式 Web 对话界面

## 三、数据层：JSON 加载与文档构建

### 3.1 原始数据结构

每条药品记录包含约 20 个字段，以下为典型示例：

```json
{
  "标题":     "新康泰克（复方盐酸伪麻黄碱缓释胶囊）",
  "通用名称": "复方盐酸伪麻黄碱缓释胶囊",
  "商品名称": "新康泰克",
  "批准文号": "国药准字H10960079",
  "药品分类": "化学药品",
  "药品性质": "处方药",
  "适应症":   "用于缓解感冒引起的鼻塞、流涕、打喷嚏...",
  "主要成份": "盐酸伪麻黄碱 120mg，马来酸氯苯那敏 8mg",
  "用法用量": "口服，每12小时1粒",
  "不良反应": "可见轻度嗜睡、口干...",
  "禁忌":     "严重高血压、冠心病患者禁用",
  "注意事项": "高血压、心脏病、糖尿病患者慎用...",
  "相关疾病": "感冒,流感,鼻塞",
  "药理毒理": "盐酸伪麻黄碱可收缩鼻黏膜血管...",
  "生产企业": "辉瑞制药有限公司"
}
```

### 3.2 三集合文档策略

**核心设计思路**：不同查询意图需要不同粒度的向量化内容。单一集合方案中，长文本会稀释名称向量，名称查询会匹配到症状语义，导致精度整体下降。因此，每条药品记录生成 **3 种不同内容** 的 `Document`，分别写入 3 个独立集合：

---

#### 集合 1：`medicine_product`（药品名称检索）

**向量化内容**（仅名称相关字段，文本极短）：
```
新康泰克（复方盐酸伪麻黄碱缓释胶囊）
复方盐酸伪麻黄碱缓释胶囊
新康泰克
xin kang tai ke
化学药品  处方药
```

**设计原因**：文本极短且高度聚焦于名称。用户输入"新康泰克"时，查询向量与此文本的余弦相似度远高于混合了大量无关内容（禁忌、用法）的长文本。名称不被其他字段"稀释"，向量空间更集中。

**补充优化**：元数据精确/包含匹配作为快速路径——先通过 ChromaDB `$eq/$contains` 直接匹配药品名，绕过向量相似度计算，彻底解决短品牌名余弦相似度偏低的问题。

---

#### 集合 2：`medicine_indications`（症状/适应症检索）

**向量化内容**（症状关键词 + 适应症全文 + 药理摘要）：
```
药品：新康泰克（复方盐酸伪麻黄碱缓释胶囊）
主治疾病：感冒 流感 鼻塞
适应症：用于缓解感冒引起的鼻塞、流涕、打喷嚏、咽喉痛及发烧等症状
主要成分：盐酸伪麻黄碱 120mg，马来酸氯苯那敏 8mg
药理：盐酸伪麻黄碱可收缩鼻黏膜血管，减轻充血水肿...
```

**设计原因**：
- 相关疾病字段拆分为空格分隔的关键词列表（`"感冒,流感,鼻塞"` → `"感冒 流感 鼻塞"`），提升关键词命中率
- 适应症全文是 BGE-M3 语义理解的主力，可将"流鼻涕"与"流涕"、"发烧"与"发热"等同义表达映射到相近向量空间
- 药理摘要（前 250 字）提供机制层语义（"收缩毛细血管" → 对应"鼻塞"症状）

---

#### 集合 3：`medicine_comprehensive`（综合查询）

**向量化内容**（名称 + 适应症 + 用法用量 + 不良反应 + 禁忌 + 注意事项）：
```
药品名称：新康泰克（复方盐酸伪麻黄碱缓释胶囊）
适应症：用于缓解感冒引起的鼻塞、流涕...
相关疾病：感冒,流感,鼻塞
用法用量：口服，每12小时1粒，不可咀嚼
不良反应：可见轻度嗜睡、口干、头晕...
禁忌：严重高血压、冠心病患者禁用...
注意事项：高血压、心脏病、糖尿病患者慎用...（前400字）
主要成分：盐酸伪麻黄碱 120mg，马来酸氯苯那敏 8mg
```

**设计原因**：文本语义丰富，能同时匹配"高血压+感冒药"、"嗜睡成分"、"儿童用药"等复杂组合查询，覆盖禁忌和注意事项等关键安全信息。

---

### 3.3 三集合对比总览

| 维度 | product | indications | comprehensive |
|------|---------|-------------|---------------|
| 向量化内容 | 仅名称/拼音/分类 | 症状关键词 + 适应症 + 药理 | 名称 + 适应症 + 用法 + 禁忌 + 不良反应 |
| 文本长度 | 极短（< 50 字） | 中等（100~300 字） | 较长（200~600 字） |
| 适用查询 | 明确药品名称 | 症状/疾病描述 | 复杂多意图问题 |
| 检索器类型 | VectorIndexRetriever + 元数据快速路径 | QueryFusionRetriever（4路 RRF） | VectorIndexRetriever + HyDE |
| 相似度阈值 | 0.55 | 0.40（RRF 分数不适用） | 0.45 |

### 3.4 文档 ID 生成与去重

```python
# 优先使用批准文号（国药准字具有法律唯一性）
if approval_no:
    return f"wenhao_{approval_no}"

# 无批准文号：商品名 + 全内容 MD5 前8位（可重现、防碰撞）
return f"hash_{name}_{md5(json.dumps(item))[:8]}"
```

三个集合使用相同 ID，确保同一药品在不同集合中可关联。进程内 `seen_ids` 集合防止重复写入。

### 3.5 文本清洗

原始 JSON 中部分字段含 HTML 标签和实体字符，清洗流程：

```python
text = html.unescape(text)                           # 解码 HTML 实体（&lt; &nbsp; 等）
text = BeautifulSoup(text, "html.parser").get_text() # 剥除 HTML 标签
text = re.sub(r"\s+", " ", text)                     # 合并多余空白
text = re.sub(r"[\x00-\x08\x0e-\x1f\x7f-\x9f]", "", text)  # 去控制字符
```

### 3.6 Metadata 与向量文本分离

Metadata 与向量化内容（`text`）独立存储：
- **向量化文本（text）**：针对检索场景优化的精简内容
- **Metadata**：存储完整的 22 个药品字段（每字段截断至 1000 字符），供 LLM 生成回答时调用
- `excluded_embed_metadata_keys`：通知 llama-index 向量化时仅使用 `text`，不将 metadata 混入向量，避免内容稀释

## 四、向量化技术详解

### 4.1 Embedding 模型：BGE-M3

| 特性 | 说明 |
|------|------|
| 模型 | `BAAI/bge-m3` |
| 向量维度 | 1024 维 |
| 最大序列长度 | 8192 tokens（本项目限制为 512） |
| 语言支持 | 多语言（中英文效果优秀） |
| 检索策略 | Dense + Sparse + ColBERT 三合一，本项目仅用稠密检索（cosine） |
| 本地路径 | `D:\Embeddings_Model\models--BAAI--bge-m3\snapshots\...` |

BGE-M3 相比同类模型的优势：

- 原生支持中文医疗领域语义，"流涕"="流鼻涕"、"发热"="发烧" 可准确匹配
- 无需微调即可在药品说明书领域达到较高召回率

### 4.2 Query 指令配置（关键优化）

BGE 系列模型对**查询文本**需加特殊指令前缀以提升检索精度，**文档文本**则不加前缀：

```
查询端：[指令] + 查询文本  →  向量 q
文档端：文档文本           →  向量 d
相似度：cosine(q, d)
```

```python
# config.py
EMBEDDING_QUERY_INSTRUCTION = "为这个句子生成表示以用于检索相关药品信息："
EMBEDDING_TEXT_INSTRUCTION  = ""    # 文档端不加前缀（BGE-M3 官方建议）

# index_manager.py
embed_model = HuggingFaceEmbedding(
    query_instruction=EMBEDDING_QUERY_INSTRUCTION,  # 手动指定（bge-m3 不在 llama-index BGE_MODELS 列表，不手动指定则返回空字符串）
    text_instruction=EMBEDDING_TEXT_INSTRUCTION,
    ...
)
```

llama-index 将指令配置到 `SentenceTransformer` 的 `prompts` 字典中，检索时自动调用 `encode(query, prompt_name="query")` 附加前缀。

### 4.3 双卡并行向量化（索引构建阶段）

**硬件**：2× RTX 5060 Ti 16GB，单卡 16GB 显存

**关键发现**：`get_text_embedding_batch(parallel_process=True)` 中的 `parallel_process` 参数在 llama-index 中**完全失效**——基类签名为 `def get_text_embedding_batch(self, texts, **kwargs)`，`**kwargs` 被接收后从不传递到底层。

**正确实现**：绕过 llama-index 包装层，直接调用底层 `SentenceTransformer.encode_multi_process`：

```python
st_model = Settings.embed_model._model        # 取底层 SentenceTransformer 实例

pool = st_model.start_multi_process_pool(
    target_devices=["cuda:0", "cuda:1"]        # 每GPU一个独立进程
)
try:
    raw = st_model.encode_multi_process(
        texts,                                 # 全量文本一次性处理，仅启动一次进程池
        pool=pool,
        batch_size=128,
        prompt_name="text",                    # 文档端不加指令前缀
        normalize_embeddings=True,             # L2 归一化（与 cosine 相似度等价）
        show_progress_bar=True,
    )
finally:
    st_model.stop_multi_process_pool(pool=pool)  # 确保进程池释放

embeddings = raw.tolist()                      # ndarray → Python list
```

全量文本只启动一次进程池，避免每个 mini-batch 重复建池的开销。

### 4.4 BFloat16 精度优化

RTX 5060 Ti 原生支持 BFloat16（指数位比 FP16 更宽，防溢出）：

```python
if torch.cuda.is_bf16_supported():
    model_kwargs["torch_dtype"] = torch.bfloat16
```

| 精度 | 显存占用 | 推理速度 | 精度损失 |
|------|---------|---------|---------|
| FP32 | ~4 GB | 基准 | 无 |
| BF16 | ~2 GB（↓50%） | 约快 20~30% | 可忽略 |

### 4.6 向量存储格式与容量

ChromaDB 使用 **HNSW（Hierarchical Navigable Small World）** 算法建立近似最近邻索引：

```python
collection = client.get_or_create_collection(
    name=collection_name,
    metadata={"hnsw:space": "cosine"},  # 余弦相似度空间
)
# 距离与相似度的转换：similarity = 1.0 - distance
```

| 集合 | 记录数 | 向量维度 | 单条占用（FP32） | 总向量占用 |
|------|--------|---------|----------------|-----------|
| product | 107,656 | 1024 | 4 KB | ~430 MB |
| indications | 107,656 | 1024 | 4 KB | ~430 MB |
| comprehensive | 107,656 | 1024 | 4 KB | ~430 MB |
| **合计** | **322,968** | — | — | **~1.3 GB** |

HNSW 查询复杂度 O(log N)，10 万条向量检索 Top-50 耗时约 20~80ms。

### 4.7 增量索引构建

`processed_ids.json` 追踪已向量化的文档 ID，新增药品数据时仅处理增量，无需全量重建：

```python
processed_ids = load_processed_ids(persist_dir)

# 仅对 new_documents 进行向量化和写入
processed_ids.update(doc.id_ for doc in new_documents)
save_processed_ids(persist_dir, processed_ids)
```

## 五、向量检索与后处理

### 5.1 三种检索增强策略

#### 策略 A：元数据精确匹配（product 集合快速路径）

```
用户输入："新康泰克"
    │
    ▼ 阶段 1：精确等值匹配（$eq）
    │  WHERE $or: [商品名称=$eq:新康泰克, 通用名称=$eq:新康泰克, 标题=$eq:新康泰克]
    │
    ▼ 阶段 2：包含匹配 fallback（$contains，Chroma 不支持 $or+$contains 组合，逐字段串行尝试）
    │  WHERE 商品名称 $contains "新康泰克"
    │
    ▼ 命中 → 直接返回，跳过向量检索（score=1.0）
      未命中 → 降级为向量检索
```

**动机**：BGE-M3 对超短品牌名（3~4 个汉字）的精确匹配能力不足，品牌名向量被长通用名稀释后余弦相似度偏低。元数据匹配完全绕过向量相似度限制，精准召回目标药品。

---

#### 策略 B：QueryFusionRetriever + RRF（indications 集合）

```
用户输入："感冒发烧"
    │
    ▼ LLM 生成 3 个中文变体查询（共 4 路）：
    │   原始：感冒发烧
    │   变体 1：感冒发热症状用药
    │   变体 2：流感退烧药推荐
    │   变体 3：上呼吸道感染治疗
    │
    ▼ 4 路分别向量化 → 检索 → 各自 Top-50
    │
    ▼ RRF（倒数排名融合）合并：score = Σ 1/(rank_i + 60)
    │
    ▼ 综合排序后取 Top-50
```

**动机**：单路查询召回率有限，中文症状表达多样（"发烧"/"发热"/"体温升高"），多路变体查询 + RRF 融合显著提升召回率。

---

#### 策略 C：HyDEQueryTransform（comprehensive 集合）

```
用户输入："高血压患者感冒了可以吃什么药？"
    │
    ▼ HyDEQueryTransform（LLM 生成假设文档）
    │  "假设药品说明书段落：...注意事项：高血压患者慎用，禁忌：
    │   严重高血压患者禁用含盐酸伪麻黄碱成分的药品..."
    │
    ▼ 平均向量 = avg(向量化(原始问题), 向量化(假设文档))
    │
    ▼ 用平均向量检索 ChromaDB
```

**动机**：问题向量（疑问句）与文档向量（陈述句说明书）在向量空间存在分布差异。HyDE 让 LLM 先生成一段"假设说明书段落"，使查询向量向文档向量空间靠近，弥合分布不一致。

---

### 5.2 后处理器链

```
Top-50 原始检索结果
    │
    ▼ [1] SimilarityPostprocessor（相似度阈值过滤）
    │     product: 0.55  comprehensive: 0.45
    │     注：indications 使用 RRF 分数（典型值 ~0.016），
    │         与余弦相似度不兼容，跳过此步，依赖 top_k 控制数量
    │
    ▼ [2] SentenceTransformerRerank（可选，默认关闭）
    │     CrossEncoder 精排（bge-reranker-v2-m3）
    │     重新计算 query-document 交叉注意力相关性分数
    │     精排后保留 Top-6
    │
    ▼ [3] LongContextReorder（结果 > 3 条时生效）
          缓解 LLM "迷失中间"问题：
          原始排列 [1, 2, 3, 4, 5, 6]（按分数降序）
          重排后   [1, 3, 5, 6, 4, 2]（高分放两端）
          LLM 对上下文开头和结尾注意力更强
```

## 六、Agent 推理层

### 6.1 三个 LangChain Tool

| 工具 | 触发条件 | 集合 | 返回数量 |
|------|---------|------|---------|
| `search_medicine_by_name` | 用户提到具体药品名称（如"阿莫西林"） | product | Top-5 |
| `search_medicine_by_symptom` | 用户描述症状/疾病（如"发烧头痛"） | indications | Top-6 |
| `search_medicine_by_comprehensive` | 复杂混合查询（禁忌/特殊人群/副作用） | comprehensive | Top-6 |

每个工具配有 Pydantic `args_schema`，规范 LLM 输入参数格式：

```python
class NameInput(BaseModel):
    medicine_name: str = Field(description="药品名称（商品名或通用名），如：新康泰克、阿莫西林")

@tool(args_schema=NameInput)
def search_medicine_by_name(medicine_name: str) -> str: ...
```

### 6.2 ReAct 推理流程（LangGraph）

```
用户输入
    │
    ▼ [Reasoning] Qwen3.5-plus 分析意图
    │  "用户问感冒发烧，需调用 search_medicine_by_symptom"
    │
    ▼ [Acting] 工具调用
    │  search_medicine_by_symptom("感冒发烧")
    │  → 向量检索 → 后处理 → 格式化文本
    │
    ▼ [Reasoning] Qwen3.5-plus 综合结果生成回答
    │  （复杂问题可能触发多轮工具调用）
    │
    ▼ 最终回答（SSE 流式 token 推送）
```

### 6.3 多轮对话记忆

`InMemorySaver` checkpointer 以 `thread_id`（UUID）隔离不同用户会话：

```python
self._checkpointer = InMemorySaver()
self.graph = create_agent(
    model=chat_client,
    tools=TOOLS,
    checkpointer=self._checkpointer,  # 每个 thread_id 独立存储消息历史
)

# 每次请求传入 thread_id，checkpointer 自动追加历史
config = {"configurable": {"thread_id": "session-uuid-xxxx"}}
result = self.graph.invoke({"messages": [("user", msg)]}, config=config)
```

服务重启后历史清空（进程内存储，非持久化）。

### 6.4 流式输出（SSE）

```python
async for event in self.graph.astream_events(input, config, version="v2"):
    if event["event"] == "on_chat_model_stream":
        chunk = event["data"]["chunk"]
        yield chunk.content              # 推送单个 token 片段
    elif event["event"] == "on_tool_start":
        yield f"\n\n> 正在调用工具：{event['name']}...\n\n"
```

### 6.5 Token 用量追踪

`TokenUsageTracker`（`BaseCallbackHandler`）记录每次 LLM 调用的 prompt/completion tokens，日志输出便于成本监控。

## 十、关键技术决策

### 10.1 为什么用三集合而不是一个集合？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 单集合（混合全量内容） | 存储简单，维护便捷 | 长文本稀释名称向量；短名称查询无法匹配症状语义；一套阈值无法兼顾三种场景 |
| **三集合（本方案）** | 各集合文本聚焦，向量空间纯净；不同意图精准路由；可为每集合单独调参 | 存储 ×3；构建时间 ×3 |

三集合方案通过 Agent 工具选择逻辑自动路由，用户无需感知内部分发机制。

### 10.2 为什么选择 llama-index 做向量化而不是 LangChain？

- llama-index 的 `HuggingFaceEmbedding` 原生支持 `query_instruction` / `text_instruction` 分离（BGE 系列核心特性）
- `VectorIndexRetriever` 与 `VectorStoreIndex` 解耦，支持 `from_vector_store()` 仅加载索引而不重新向量化
- `StorageContext` 统一管理向量存储后端，切换 ChromaDB / Faiss / Qdrant 只需修改一处

### 10.3 为什么自定义 ChromaVectorStore？

`llama-index-vector-stores-chroma` 包对 chromadb 1.5 存在 API 兼容性问题。自定义适配器直接调用 `chromadb.Collection` 原生 API，同时实现了：
- MetadataFilters → Chroma `where` 子句（12 种 FilterOperator 完整映射）
- MMR（最大边际相关度）查询模式（相关性与多样性平衡）
- 空 metadata 自动补位（chromadb 1.3+ 新增非空校验）

### 10.4 为什么用 InMemorySaver 而不是数据库持久化？

当前单服务器部署场景不需要跨进程或重启后的历史恢复，`InMemorySaver` 零依赖、零延迟。若后续需要持久化，LangGraph 支持替换为 `SqliteSaver` 或 `PostgresSaver`，仅需修改一行代码。
